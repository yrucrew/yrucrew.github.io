1.1.5. Processing

DispatcherServlet 的请求流程如下：

  • WebApplicationContext
  • locale resolver（本地化处理器）
  • theme resolver
  • multipart resolver
  • 寻找一个适当的 handler。如果找到则把 handler（前置处理器，后置处理器以及 controllers）关联到执行链，准备 model 进行渲染。另外，对于带注解的 controller，可以渲染 response （within the HandlerAdapter），而不是返回 view。
  • 如果返回 model，则渲染 view。如果没有返回 model（可能是由于预处理器或后处理器拦截了请求，也可能是出于安全原因），则不会渲染 view，因为请求可能已经完成了。

1.1.8. Exceptions

通常情况下，错误响应如果在请求映射期间发生异常或从请求 handler（如 @Controller ）抛出异常，DispatcherServlet 将委托给 HandlerExceptionResolver beans 链来解析异常并提供替代方案。

Chain of Resolvers

可以在 Spring 配置中声明多个 HandlerExceptionResolver beans，按需要设置相应的 order。The higher the order property, the later the exception resolver is positioned.

1.3. Annotated Controllers

Spring MVC 提供了基于注解的编程模型，@Controller 和@RestController 使用注解来表示请求映射，请求输入，异常处理等等。注解控制器拥有灵活的方法签名，不需要继承基础类或者实现特定的接口。

1.3.1. Declaration

@RestController 是一个组合注解，继承了@ResponseBody ，因此会直接写响应体而不是视图或者渲染 HTML 模板。

AOP Proxies

某些时候，需要再运行时使用 AOP 代理装饰 controller。例如直接在 Controller 上使用 @Transaction 注解，类似这种情况，官方推荐使用基于类的代理，这通常也是 controllers 的默认行为。但是，如果控制器必须实现一个不是 Spring Context 回调的接口（例如 InitializingBean、*Aware等），您可能需要显式地配置基于类的代理，使用 @EnableTransactionManagement 的话修改成 @EnableTransactionManagement(proxyTargetClass = true)
